# GlyphLens 生态调研与下一步规划（2026-04-24）

> 目的：扫一遍 GitHub 上"文物 / 古文字 / 碑刻 / 拓片 / 古籍"方向的开源生态，定位 GlyphLens 的差异化，给出可操作的下一步思路。
> 前序：`开发进展-20260424.md`（字迹标注重写 + 区域 OCR 已落地）。

---

## 1. 当前 GlyphLens 的能力边界

已有：
- 图像处理管线：原始 / 微痕 / 锐化 / 灰度 / 数字线图 / 拓片（六产物，参数化 + 细线描边）
- 字迹标注：自绘视口，归一化 bbox，底图可切换对齐
- 区域 OCR：后端 `rapidocr` 裁切识别
- 数据管理：文物 CRUD、管线重跑

定位关键词：**碑刻 / 画像石 / 墓志铭**的**研究型释读工具**（重"处理 + 标注 + 复核"，轻"端到端识别"）。

## 3. GlyphLens 的差异化定位

对比上面的项目，我们可以明确：

- **不是**又一个端到端 OCR 模型（PP-OCR / 甲骨识别模型已经做得很好）
- **不是**通用 IIIF 标注器（Annotorious / IMMARKUS 已经做得很好）
- **而是**：为**碑刻 / 画像石**研究场景定制的、**以"人机协同释读"为核心**的工作台
  - 人看不清 → 我们把图处理得能看清（**已有**，且可调）
  - 人框字费劲 → 我们预切单字 + OCR 候选（**下一步**）
  - 人要校对留档 → 我们保证归一化坐标、底图切换、导出格式（**已有**）
  - 人要写释文 → 我们接字典 / 断句 / 五体对照（**中期**）

## 4. 下一步思路（按优先级）

### P0 · 短期（1~2 迭代，工程已就位）

| 功能 | 借鉴对象 | 实现要点 |
|---|---|---|
| **单字自动检测 → 预标注** | `LIU42/Inscriptions` | 后端新增 `/api/auto-detect`，用 YOLOv8 或 `rapidocr` 自带的 det 模型，输出归一化 bbox 列表，前端批量灌入为"待审 draft" |
| **OCR 候选字下拉** | PP-OCR 流式 | 现在 `rapidocr` 已返回文本；补一个"top-K 候选"接口，表单 `glyph` 字段改为可选下拉 |
| **标注导出兼容格式** | PPOCRLabel | 现在已有 JSON，**再加一种 `label.txt`（PP-OCR 格式）** 和 COCO JSON，方便送外部训练 |
| **读序（reading order）** | ndlkotenocr | 对已标注框做简单启发式排序（从右到左、从上到下），给每个 bbox 存一个 `order` 字段，列表按此排 |

### P1 · 中期（需要引入新依赖或少量模型）

| 功能 | 借鉴对象 | 实现要点 |
|---|---|---|
| **字形对照面板** | `makemeahanzi` + `calligraphy-community` | 释读字段右侧弹出小面板：显示该字的篆/隶/楷等字形参考（静态字形库就够用） |
| **释文自动断句 + 标点** | `Jiayan` | 后端新增 `/api/punctuate`，输入按 order 拼好的释读字符串，返回带标点版本 |
| **字源注（《说文》）** | `shuowenjiezi/shuowen` | 生僻字 hover 时调字典接口显示字源 |
| **多文物数据集管理增强** | `Open-Oracle` 组织方式 | 现在按单件文物组织；加"项目/合集"一层，方便把一组造像石当作一个数据集 |

### P2 · 长期（偏研究，需模型训练）

| 功能 | 借鉴对象 | 技术栈 |
|---|---|---|
| **残字补全** | `RubbingGAN` / `OBSD` | 选中残缺框 → GAN/扩散模型推荐补全字形，人工确认 |
| **释文序列修复** | `ancient-text-restoration` (Pythia) | 已有部分释文 → LM 推测缺失字，类似"完形填空" |
| **五体字迹识别** | `jfxia/shufa` | 给一个字形图，自动判别篆/隶/楷，辅助断代 |
| **多人协作 + 版本** | `liiive` / `guji-toolkit-web` | SQLite → PostgreSQL，加用户表 + WebSocket 实时同步 |

## 5. 不建议做的方向（避免踩坑）

- ❌ **从零训练通用 OCR 模型**：PP-OCR / rapidocr 已经够用，我们的价值在前端处理和人机协同
- ❌ **换成 IIIF / OSD 生态**：已经主动去掉 OSD 了，自绘视口更可控且满足需求
- ❌ **甲骨文专题**：跟当前"碑刻/画像石"定位不符，别分散精力；要做等核心打磨完再开支线

## 6. 技术落地建议（给 P0 四件事排顺序）

按改动量从小到大：

1. **标注导出兼容格式**（纯前端，1~2 小时）
   - `AnnotationModule` 加"导出"按钮，格式下拉：GlyphLens JSON / PP-OCR label.txt / COCO
2. **读序**（前后端各一点，0.5 天）
   - 后端给 `annotations` 表加 `order_index` 字段（可空）
   - 前端列表加"重排序"按钮，默认按启发式规则（右→左，上→下，y 容差 30%）
3. **OCR top-K 候选**（纯后端增量，0.5 天）
   - `rapidocr.run()` 结果本来就带 score，把 `rec_boxes` 相关的 top-K 透传给前端即可
4. **单字自动检测**（需要模型，1~2 天）
   - 先**别**自训模型；直接复用 `rapidocr` 的 det 模型做"文字区域粗检测"
   - 若效果不佳，再引入 `LIU42/Inscriptions` 的 YOLOv8 权重（MIT License，可直接用）
   - 前端"一键预标注"按钮：调接口 → 返回 N 个 bbox → 批量落为 draft，用户逐个审核 glyph
